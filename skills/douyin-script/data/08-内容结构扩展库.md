# 内容结构扩展库

> 来源：内容框架与结构研究结果 + 7步创作流程

---

## 一、9种通用内容结构

当获客型四个核心模板不够用时，从以下扩展库中选择：

| 结构类型 | 公式 | 适用场景 |
|----------|------|----------|
| 观点+事例型 | 核心观点 → 多个故事印证 → 升华 | 生活感悟、社会观察 |
| 大观点+小观点型 | 打破常识的大观点 → 多个小观点论述 | 认知颠覆类、热点评论 |
| 干货教学型 | 问题场景 → 步骤拆解 → 成果展示 | 知识付费、技能培训 |
| 痛点共鸣型 | 痛点描述 → 情绪共鸣 → 解决方案 | 消费决策、情感内容 |
| 悬念反转型 | 反常事件 → 悬念留白 → 反转结局 | 搞笑、情感、剧情 |
| 对比反差型 | 通过对比展现差异 → 引发兴趣 | 测评、种草、前后对比 |
| 挑战互动型 | 规则说明 → 示范动作 → 点名互动 | 美妆、健身、美食 |

**注意**：获客型内容优先使用 data/01 的四个模板。扩展库里的结构只在默认模板不适用时作为替代。

## 二、推力结构是底层技法，不是独立结构类型

推力结构（悬念递进、信息密度控制、情绪波动）可以叠加到任何结构上：

- 观点+事例型 → 在每个事例之间用悬念递进衔接
- 干货教学型 → 用信息密度控制保证每15秒有新信息点
- 痛点共鸣型 → 用情绪波动法保持节奏

## 三、5步逆向拆解法（分析对标用）

当用户提供对标内容时，用它拆解：

1. **扫描整体框架**：诉求目标/目标受众/核心逻辑
2. **提取关键模块**：开头（钩子）→ 中间（证据/优势）→ 结尾（CTA/升华）
3. **模式匹配**：把拆出的模块跟已知结构对比
4. **处理混合结构**：逐段标注功能标签，找出主导模式
5. **验证**：读出声检查流畅度，问这个结构是否解决了用户的问题

## 四、开头钩子四维拆解法

拿到对标开头，拆四样东西：
1. 她说了什么？（字面内容）
2. 她用的是哪种钩子类型？（提问/反常识/痛点/数字冲击/悬念）
3. 她针对的是哪种情绪？（焦虑/好奇/共鸣/渴望）
4. 她用的句式是什么？（能不能抽象成一个公式？）

**示例**：
- 原文："我工作五年，攒下的钱还不如我同学炒一次基金"
- 钩子类型 = 反差对比
- 情绪 = 共鸣 + 轻微焦虑
- 句式公式 = `「我付出XX，却比不上别人XX」`

同一个博主通常只反复用2-3种开头公式，拆出来就是可复用的模板。

## 六、观察流：安先生式高密度感知写作法

### 核心区别

普通获客稿在**论述**——"我认为X，因为Y，所以你应该Z"。
观察流在**呈现**——"你看这个东西是这样的，你看那个人是那样的，它们之间的裂缝就是阶级。"

论述流让用户觉得"这个人说得对"。
观察流让用户觉得"这个人看到的世界跟我看到的不一样，我想借他的眼睛"。

### 底层机制

观察流信任用户自己的推导能力。你把细节给够、把尺度拉大、把对照做清楚——用户自己会把结论推出来。自己推出的结论，比被告知的结论牢固十倍。

这也是为什么观察流天然适合获客：用户推导出"我需要换一种获取财富的方式"之后，他的下一步需求就是"谁能帮我换"——而不是"我自己再琢磨琢磨"。

### 五种核心技法

#### 1. 细节锚定：每个抽象概念必须落地为一个可感知的细节

坏的写法："挣的人很焦虑，被时间压迫。"

好的写法："挣的人走路很快。他不是天生急性子，是他脑子里有一个计时器在走。跟任何人说话，说不到三分钟就开始看手机上的时间。"

规则：
- 不直接命名情绪（焦虑、恐惧、渴望）
- 用身体状态命名情绪（走路快、手指紧、语速加快、看表的频率）
- 宁可多写一个细节，不少写一个概念

#### 2. 尺度跳跃：在个体/群体/城市/社会/语言之间来回切换

安先生稿子的尺度切换路径：
```
语义拆字（挣 vs 赚）
→ 个体身体（手指松紧、走路速度）
→ 群体空间（地铁、咖啡馆）
→ 个体消费行为（买包、发朋友圈）
→ 城市面貌（夜宵摊 vs 咖啡馆的分布）
→ 回归语义（两个汉字的终极对照）
```

规则：
- 每 2-3 段必须切换一次尺度
- 同一尺度不连续出现两次
- 最大跳跃（个体→社会）和最小跳跃（个体→群体）交替使用
- 起点和终点用同一个锚点意象（如拆字），形成闭环

#### 3. 自我退场：观察者不出现

安先生的稿子里没有任何"我做过X""我一天工作X小时""我的客户说"。

信用不来自自述经历，来自观察质量。当你说出一个用户无数次感觉到但从未说清楚的细节时，你不需要证明自己是谁。

规则：
- 可以用"你看"引导，但不说"我觉得"
- 可以引述第三方——"你看咖啡馆里那些做投资的人"——但这是群体观察，不是自述
- 如果选题必须用个人经历支撑，把经历写成"有一类人"或"你去看X群体"，而不是"我"
- 例外：选题本身是"记录真实过程型"（如30天起号记录），才适合出现自我

#### 4. 结尾留白：不给答案，给一个更大的问题

安先生结尾："那个鸿沟是很难跨越的。"
本篇结尾："至于怎么换过来——那是另一个问题了。"

规则：
- 不总结、不升华、不"所以你应该"
- 用一个观察收束，或抛回一个问题
- 结尾的质量 = 用户放下手机之后还在想的时长
- 如果用户看完立刻关掉 → 结尾太封闭
- 如果用户看完打开评论区 → 结尾对了

#### 5. 价值种入：在问题与桥接之间，植入解决方案的可观察形态

观察流最容易犯的错误是：问题呈现了、桥接也做了，但中间漏了一环——用户不知道**解法长什么样**。

他不知道"系统"具体是什么。IP？Token？轻创业？一个人能跑的项目？这些不是执行细节，它们是**价值判断**——告诉用户"这个时代存在这样的选择"，不等于教他"怎么搭Token"。

**价值种入的位置**：在问题观察完成之后、结尾桥接之前。结构是：

```
问题观察（挣的人什么样）
→ 问题深化（消费暴露/城市面貌）
→ 价值种入（解法长什么样——IP/Token/轻创业的可观察形态）
→ 回归锚点（回到挣 vs 赚）
→ 结尾桥接（我见过换过来的人）
```

**价值种入的写法**：
- 保持观察流姿态——"你看那些做IP的人""你看那些做轻创业的"
- 描述的是**形态**而非**操作**——"一个账号搭好内容框架，自己在那替你说话"（形态），不是"先注册账号，然后定位赛道，然后做选题矩阵"（操作）
- 每个形态用一个"你看"引导，继续用第三人称观察
- 种入 2-3 个形态即可，不是产品列表

**价值种入 vs 产品推销的区别**：
| 价值种入（应该做） | 产品推销（禁止做） |
|---------|---------|
| "你看那些做IP的人，一个账号搭好框架，它自己在那替你说话" | "我做IP孵化，帮你搭账号" |
| "你看Token，代码替你执行规则，你不在场结算照样完成" | "我有Token项目，你投钱我帮你操作" |
| 第三人称，群体观察 | 第一人称，自我推销 |
| 用户感觉：原来还有这种选择 | 用户感觉：这人要卖我东西 |

**价值种入后的获客链路**：
用户看到了问题 → 用户看到了解法形态 → 用户知道"这种解法存在，但我不会搭" → 结尾桥接"我见过换过来的人" → 点进主页 → 发现你就是提供这种解法的人

### 观察流适用判断

| 选题类型 | 适合观察流？ | 原因 |
|---------|------------|------|
| 社会阶层/文化批判 | 最适合 | 观察本身就是论点，细节就是论据 |
| 行业公敌/反常识颠覆 | 适合 | 观察流让"攻击旧模式"更有杀伤力——你不是在喊口号，你是在指给你看 |
| 个人经历/案例论证 | 部分适合 | 把"我"退场，案例写成"有一类人" |
| 趋势断言/未来判断 | 不太适合 | 论断需要明确的观点陈述，观察流太含蓄 |
| 干货教学/步骤拆解 | 不适合 | 干货需要直接给框架，观察流会模糊重点 |

### 观察流 × 七种切换技法

观察流天然要求高频切换，但切换的驱动方式不同：

| 技法 | 论述流怎么用 | 观察流怎么用 |
|------|------------|------------|
| 认知层级 | 难易交替论证 | 从细节中自然浮现认知，不直接说结论 |
| 视角切换 | 我/你/他交替 | 观察者（你看）+ 被观察者（他）+ 观众（你）三重视角 |
| 时间轴 | 过去/现在/未来 | 此刻的场景 → 十年前的对比 → 十年后的推演 |
| 情绪温度 | 冲击→理性→兴奋 | 冷静的好奇→细节的共鸣→沉默的震动 |
| 颗粒度 | 概念→案例→总结 | 字体笔画→手指→车厢→城市→社会结构（自动形成） |
| 论证模式 | 逻辑/故事/数据交替 | 用呈现替代论证——一个细节就是一个论据 |
| 对话感 | 模拟对话互动 | 用"你看"邀请用户共同观察，全程对话姿态 |

### 常见错误

1. **细节堆砌而非细节选择**：不是把所有细节都写进去。选那个最反直觉的、最有阶级暴露感的、用户最可能从来没注意到的。
2. **观察完之后忍不住总结**：信任你的细节。如果细节够好，总结是多余的。
3. **观察者变成了偷窥者**：观察是"我带你一起看"，不是"我看了然后告诉你"。姿态是平等的。
4. **自我偶尔冒出来**：写了三个细节之后突然来一句"我就这样做了三年"，整段观察的信任感瞬间崩塌。
5. **把"不提执行细节"理解成"不提解决方案的存在"**：这是观察流在获客场景里最致命的错误。IP、Token、轻创业项目——这些是价值判断（"这个时代存在这样的选择"），不是执行交付（"怎么搭Token"）。一刀切砍掉的结果是用户看完知道问题但不知道有解法，获客链路断裂。正确做法：用观察流呈现解法形态（"你看那些做IP的人..."），不教操作步骤。

### 结尾桥接：观察流获客的最后一步

观察流的代价是自我退场——用户被你的观察打动，但不知道正在跟他说话的人就是能帮他的人。结尾需要在不破坏观察姿态的前提下，架一座隐形的桥。

**差的结尾（桥断了）**：
- "至于怎么换过来——那是另一个问题了。" → 用户觉得"好吧，我回去想想"。没有下一步动作。
- "欢迎大家评论区讨论。" → 评论区是讨论了，没人来找你。

**好的结尾（隐形桥）**：
- "至于怎么换过来——我见过换过来的人是什么样的。下次讲。" → 预告下期 + 暗示"我认识路径"，不破观察姿态

**桥接规则**：
1. 桥接用"我见过"而非"我做过"——保持观察者身份，不突然变成经历自述
2. 桥接指向"下一期内容"而非"找我咨询"——不破 CTA 禁令，但制造追看
3. 桥接内容必须是真实的——如果你没见过换过来的人，不能编
4. 桥接位置在全文最后一句，替换纯留白式结尾

**桥接后用户的推导链**：
这个人观察力很强 → 他见过换过来的人 → 他可能认识路径 → 点进主页看看他还讲过什么 → 发现产品/服务 → 私信

**什么时候不桥接**：
- 用户体量已经够大，主页流量自然充足 → 不需要
- 选题本身和产品关联极弱 → 桥接了也是断的
- 结尾有更好的处理方式（如金句收束比桥接更有力） → 不勉强
