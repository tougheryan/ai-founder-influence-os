# 获客型抖音口播稿生产系统

一个 Claude Code Skill —— 渐进式个人获客内容操作系统。每次对话都有产出，每次产出都有沉淀。

## 这是什么

不是泛流量口播稿生成器。这个 Skill 专做一件事：**让观众看完你的视频后主动来找你开口说话**。

核心逻辑来自 13 章获客型文案白皮书（6 万字方法论），所有内容基于**你的真实经历和产品**，不编造、不套模板。

## 设计哲学

- **轻入口**：第一次使用不需要完整知识库，有一句话就能开始
- **强恢复**：每次新窗口强制展示"获客内容作战台"，知道做到哪了、该做什么
- **模块化路由**：8 个模块各自独立，按需激活，不绑架用户走完流程
- **渐进式积累**：每次对话顺手收集信息，知识库越用越厚，稿子越来越不可替代

## 触发方式

**精准触发**（指向获客系统）：
- `/douyin-lead-gen`、`/获客口播`
- 「帮我写获客稿」「获客型内容」「让人主动私信的稿子」
- 「为什么这篇没人咨询」「把这个选题改成能获客的口播」
- 「生成获客选题」「用行业选题连连看生成选题」

**弱化触发**（优先交给泛流量系统）：
- 「写口播」「写抖音口播稿」→ 默认走 `douyin-script-maker-v2`

## 模块化路由

每次触发先展示 **Context Boot（获客内容作战台）**，然后自动判断进入对应模块：

| 模块 | 触发场景 | 做什么 |
|------|---------|--------|
| **A: Context Boot** | 每次新窗口 / "继续上次" | 强制展示作战台：定位 + 进度 + 推荐下一步 |
| **B: 选题生成** | 没有选题 / "今天拍什么" | 行业选题连连看：25 格矩阵生成 20-30 个候选 |
| **C: 选题转化** | 有普通选题但不够锋利 | 改成获客型选题，输出 2-3 个角度版本 |
| **D: 快速成稿** | 有选题 + 要成稿 | 分层读取，只问 1-3 个问题，1000-1400 字 |
| **E: 审核修改** | 有旧稿要诊断 | 获客链路诊断 + 评分 + 可选直接输出替换稿 |
| **F: 深度建库** | "帮我系统建库" | 7 文件完整定位梳理（非默认入口） |
| **G: 数据复盘** | "数据回来了" | 平台数据 → 获客诊断 → 知识库校准 |
| **H: 知识库回流** | 每次任务结束后静默执行 | 提取新增信息回写知识库，不打断主任务 |

## 行业选题连连看

当用户没有选题时，不卡住、不追问。直接用 25 格选题矩阵生成：

- **行业核心**（1 个）：用户的行业/产品
- **行业相关**（8-12 个）：细分领域、常见问题
- **人群/场景**（10-15 个）：会接触这个行业的人或场景

三点一线组合：**行业核心 + 行业相关 + 人群/场景** → 20-30 个候选 → 获客筛选 → 5 个最值得写的选题。

## 核心铁律

- **价值讲透，执行留白** — 讲判断框架、底层逻辑、错误路径、价值边界。不免费交付 SOP、工具配置、执行清单
- **禁止在结尾放 CTA** — 不写"私信我""加我微信"
- **不编造经历** — 知识库里没有的素材，停下来问用户
- **禁止推销员语气** — 你是行业规则的制定者，不是卖货的
- **边用边长** — 第一次不需要完整知识库，每次对话顺手收集

## 评分系统

评分是诊断工具，不是预测结果。30 分以上只代表结构具备发布条件，不代表一定能获客。

每次评分输出：结构分 + 获客链路风险 + 最大不确定性 + 下一条要验证的假设。

不足 10 条历史样本前，不输出过度确定的预测。
